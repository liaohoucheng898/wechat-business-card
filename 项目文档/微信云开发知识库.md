## 一、怎么使用这份知识库

这份文档不是原文搬运，而是把官方文档按“真实开发最有用”的方式重新整理成了三层：

1. 先看“铁律”
   - 这是以后做云开发时必须先记住的规则。
2. 再看“能力分区”
   - 包括环境、数据库、存储、云函数、Web、权限、安全、运维等。
3. 最后看“覆盖页目录”
   - 这里列出了本次已覆盖阅读的官方页面，方便继续追溯原文。

## 二、先记住的 20 条铁律

1. 正式环境和测试环境必须分开，至少一套 `test`，一套 `release`。
2. 小程序端 `wx.cloud.init` 只决定客户端默认环境，不会自动决定云函数里的环境。
3. 云函数里必须再次 `cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })`，不要写死环境。
4. 客户端能做的数据库操作，只适合非敏感数据；敏感操作必须放云函数。
5. 不要把业务安全建立在前端按钮隐藏上，权限要在云函数里判。
6. 未登录模式默认不安全，必须显式开启，并且必须配安全规则。
7. 数据库、存储、云函数三类资源，面向 C 端时优先用安全规则，而不是简易权限。
8. 云函数是无状态的，不要指望上一次调用留下的内存或磁盘还在。
9. 云函数 `/tmp` 只有临时价值，不是持久存储。
10. 云函数运行时区默认是 UTC+0，不是北京时间。
11. 存云存储时，数据库里保存永久 `fileID`，不要保存临时 `tempURL`。
12. `tempURL` 只是展示用的短期地址，用时再换。
13. 富文本、图片、文件上传都要考虑内容安全和权限边界。
14. 小程序端分享、Web 端访问、未登录访问，本质上是三条不同安全链路，不能混着想。
15. 数据库规模一大，`count()`、全表扫描、逐人循环查询会立刻变慢。
16. 有列表、排行、趋势统计时，要提前设计聚合表或归档表。
17. 所有验证码逻辑都要同时绑定：手机号、场景、有效期、是否已用、错误次数、锁定时间。
18. 所有密钥、票据、私钥都不能进仓库，要走部署注入。
19. 运行时外链 CDN 要谨慎，能本地化就本地化。
20. 做云开发项目时，代码之外至少还要同步维护：权限配置、索引、告警、备份、发布回滚说明。

## 三、基础概念整理

### 1. 云开发到底是什么

官方文档强调，云开发不是单独的“云函数服务”，而是一整套微信生态内的一体化后端能力，核心包含：

1. 云数据库
2. 云存储
3. 云函数
4. 静态网站托管 / Web SDK / 云调用
5. 数据模型、CMS、扩展能力、工作流、低码能力

对当前这个项目来说，最核心的是前 3 项，再加上：

1. 内容安全
2. 图片处理
3. 统计归档
4. Web / 管理端接入

### 2. 环境

官方文档里对“环境”的强调非常重，这也是最容易被新手忽略的地方。

你要把环境理解成“一整套独立资源空间”，它不是一个简单变量，而是整套后端隔离单元。一个环境通常包含：

1. 独立数据库
2. 独立存储
3. 独立云函数
4. 独立权限与配置
5. 独立计费与告警

官方建议每个正式环境都搭一个测试环境，这一点必须执行。  
本项目后续也应该坚持：

1. `test` 环境做调试、验收
2. `release` 环境做正式运行

### 3. 账号与控制台

官方文档说明，微信云开发和腾讯云控制台并不是“一个账号体系”直接等价。  
实际开发里要记住：

1. 小程序管理员拥有最高权限
2. 云开发管理员可由小程序管理员指定
3. 云开发开发者权限需要在控制台细分配置

这意味着：

1. 代码里有权限校验不够
2. 控制台里也要配正确的角色
3. 如果控制台权限没配，开发者可能默认权限过大

### 4. 未登录模式

官方文档明确说了，未登录模式不是默认可用的，它必须显式开启。并且一旦开启：

1. 必须配安全规则
2. `auth == null` 要视为未登录访问
3. 数据库、存储、云函数都要分别考虑未登录访问边界

这对当前项目非常关键，因为：

1. 客户查看名片页，本质就是公开访问场景
2. 公开接口一定要严格限制返回字段
3. 公开访问不能顺手把后台或内部字段带出来

### 5. 权限设置

官方文档给的思路很明确：

1. 小程序管理员负责最高层授权
2. 云开发管理员负责环境级管理
3. 云开发开发者只拿完成工作所需的最小权限

落地到项目里就是：

1. 代码权限判定要最小化
2. 控制台权限也要最小化
3. 不要让“会进开发者工具的人”默认等于“能全管后端的人”

### 6. Web / 公众号 / 跨账号环境共享

官方文档说明，Web 端、公众号、环境共享都能接云开发，但它们不是“小程序端换个壳”那么简单。

必须额外注意：

1. Web SDK 的登录态处理与小程序不同
2. 第三方 Cookie 限制会影响公众号场景
3. 跨账号环境共享时，要明确资源方、调用方、身份边界

如果以后本项目要继续扩 Web 管理能力，这部分必须提前设计，而不能等出问题再补。

## 四、开发指引整理

### 1. 控制台、初始化、调试

#### 控制台

官方文档把控制台视为“代码之外的半个后端”。这点很对。  
真正能跑稳的项目，控制台里至少要维护：

1. 环境
2. 权限
3. 数据库集合
4. 索引
5. 存储规则
6. 云函数版本
7. 日志与告警

#### 初始化

官方强调两件事：

1. 小程序端 `wx.cloud.init` 只能调一次
2. 云函数端要单独初始化环境

对项目的直接落地规则：

1. 小程序 `app.js` 里统一初始化
2. 云函数里统一写 `cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })`
3. 禁止在正式代码中写死测试环境 ID

#### 调试

官方文档提醒可以从开发者工具和 Network 面板看当前请求使用的是哪个环境。  
以后排查“为什么测试数据跑到正式环境去了”这种问题，第一件事就是看初始化时到底落到了哪个环境。

### 2. 数据库

#### 官方能力总结

官方数据库是 JSON 文档数据库，支持：

1. 基础增删查改
2. 查询指令
3. 数组 / 对象操作
4. 联表查询
5. 聚合
6. 事务
7. 地理位置
8. 实时推送
9. 管理端脚本
10. 数据迁移、导入导出、备份

#### 做项目时最重要的 10 条数据库规则

1. 客户端只做低风险读写，核心写操作走云函数。
2. 设计集合前先想索引，不要等慢了再补。
3. 排行、趋势、统计类接口优先做汇总表。
4. 少做全量扫表，多做条件过滤和分页。
5. 有并发修改风险的地方要做版本或时间戳保护。
6. 安全规则优先，不要长期依赖简易权限。
7. 富文本、长文本、用户输入内容要做内容安全和净化。
8. 历史日志要归档，不要永远堆在主表里。
9. 批量管理、导入导出、脚本改数都要有回滚方案。
10. 改查询前先跑 Explain，确认有没有命中索引。

#### 安全规则

官方数据库安全规则是做 C 端安全边界的重点能力。  
如果一个项目会对外公开访问，就应把规则设计清楚：

1. 谁能读
2. 谁能写
3. 能读哪些字段
4. 能写哪些字段
5. 未登录是否可访问

本项目以后最该补的不是“更多 if 判断”，而是数据库与存储规则跟代码一起收口。

#### 事务

官方明确支持事务。  
凡是下面这些场景，都要优先考虑事务或至少保证幂等：

1. 先改员工，再改管理员配置
2. 先写主表，再写统计表
3. 先验证，再扣减 / 增加计数

#### Explain 与索引

官方文档给了 `Explain` 查询分析，这个能力以后要常用。  
凡是下面这些接口，一旦慢，就先查 Explain：

1. 员工列表
2. 数据统计
3. 浏览排行
4. 案例筛选
5. 日志归档

### 3. 存储

#### 官方能力总结

云存储支持：

1. 上传
2. 下载
3. 管理文件
4. 安全规则
5. 文件命名限制
6. 组件支持

#### 项目落地规则

1. 数据库存 `fileID`，不存临时链接。
2. 展示时再换 `tempURL`。
3. 文件路径要有业务前缀，例如：
   - `avatars/`
   - `admin-upload/editor/`
   - `cases/`
4. 上传前限制格式、大小、尺寸。
5. 后台删除业务记录时，要考虑是否同步清理无用文件。
6. 存储权限不能默认全开。

#### 临时链接

官方模式决定了：  
`tempURL` 是临时访问地址，不可长期保存、不可当永久资源地址。  
这也是本项目之前富文本图片显示异常的根本背景之一。

### 4. 云函数

#### 官方能力总结

云函数支持：

1. 小程序端调用
2. 云函数互调
3. `wx-server-sdk`
4. npm 依赖
5. 本地调试
6. 版本管理 / 灰度
7. 日志与监控
8. 定时触发器
9. 事件触发器
10. 常驻云函数
11. 云调用

#### 运行机制必须牢记

官方明确强调：

1. 云函数运行在云端 Linux
2. 实例之间互相隔离
3. 会自动扩缩容
4. 是无状态的
5. `/tmp` 只有 512MB，且可能随时消失
6. 默认时区是 UTC+0

换成项目语言就是：

1. 别把状态放内存里指望下次还在
2. 别把重要文件留在 `/tmp`
3. 别把时间计算默认当北京时间
4. 别依赖本地 Windows / macOS 行为去推断云端结果

#### 云函数安全规则

官方文档强调，云函数对 C 端用户访问也可以配安全规则。  
这意味着：

1. 并不是“都走云函数就自然安全”
2. 公开可调的云函数仍要配访问边界
3. 未登录模式下尤其要控制可调用函数范围

#### 云函数开发注意事项

官方“注意事项”对实际项目非常重要：

1. Node 原生依赖要考虑 Linux 兼容
2. 少依赖系统命令
3. 时区要主动处理
4. 第三方包要控制版本
5. 云函数最好幂等

#### 本项目落地规则

1. 统一用一套返回结构：`code / msg / data`
2. 鉴权逻辑抽共享模块
3. 敏感日志绝不输出验证码、私钥、完整手机号
4. 环境变量和密钥不进仓库
5. 定时任务统一写清楚触发频率、归档范围、失败告警

### 5. Web SDK、公众号、第三方 Cookie

官方文档的核心结论：

1. Web SDK 可访问云开发资源
2. 公众号场景下，登录态可能受第三方 Cookie 限制
3. iOS 14 以后，很多场景必须改走第一方域名代理

如果以后要给本项目再加公众号或 Web H5：

1. 先设计登录态，不要直接照抄小程序
2. 先确认域名与 HTTPS
3. 先确认第一方 Cookie / 代理方案

### 6. 云调用

官方文档说明，云调用可以让云函数直接调用微信开放接口，不需要开发者自己管 `access_token` 换取流程，但有两个前提：

1. 这个接口本身支持云调用
2. 云函数目录下的 `config.json` 要声明对应权限

以后如果项目加：

1. 订阅消息
2. 模板通知
3. 支付回调后推送
4. 审核 / 内容接口

都应该先看云调用支持情况，再决定技术路径。

### 7. 内容安全

官方文档给了云开发内容安全能力，核心价值不是“可选插件”，而是：

1. 降低审核失败风险
2. 降低线上违规风险
3. 给数据库文本字段自动做违规检查和改写

本项目已经有公司介绍、业务介绍、案例详情等富文本内容，后续强烈建议：

1. 关键富文本字段接内容安全策略
2. 上传图片也走安全检查
3. 保存前做本地校验，保存后做服务端审核

### 8. 图片处理

官方图片处理能力的核心思路是：

1. 原图存一份
2. 展示时按样式实时裁切 / 缩放
3. 不一定非要二次写回数据库

对本项目的启发：

1. 名片头像、案例封面、富文本插图都可以统一出一套展示规格
2. 如果用官方样式处理，能减少前端到处手写裁切逻辑
3. 但要先明确计费和样式命名规范

### 9. 数据模型、CMS、工作流、扩展能力

这些是官方在云开发上的更高阶能力，不是当前项目必须马上上，但以后扩项目会很有用：

1. 数据模型：适合把结构、校验、关联统一起来
2. CMS：适合内容运营
3. 云后台：适合快速搭管理系统
4. 工作流：适合审批、通知、自动处理
5. extend.AI / Agent：适合做智能助手和智能流程

对本项目的现实建议是：

1. 当前仍以“手写云函数 + 手写前端”最稳
2. 以后如果内容运营需求增长，再考虑 CMS / 数据模型
3. 以后如果审批链增加，再考虑工作流

### 10. 微信支付、资源管理、运维

官方文档告诉我们，云开发不只是“能写接口”，还包括：

1. 支付接入
2. 配额与计费
3. 告警
4. 仪表盘监控
5. 工单

项目长期要稳，必须把“运维”视为开发的一部分：

1. 给关键云函数配置告警
2. 监控错误次数和耗时
3. 定期看资源消耗
4. 大版本前留备份和回滚包

## 五、针对本项目的落地规范

### 1. 环境规范

1. 正式环境和测试环境分离。
2. 云函数统一使用 `cloud.DYNAMIC_CURRENT_ENV`。
3. 前端配置只放环境 ID、公开地图 Key 一类信息，私密信息走部署注入。

### 2. 鉴权规范

1. 小程序 C 端用户态：`sessionToken`
2. 管理端：CloudBase Custom Login
3. 管理权限必须后端二次校验，不只看前端页面
4. 验证码逻辑必须绑定 `scene`

### 3. 数据库规范

1. 主业务表：
   - `staff`
   - `companies`
   - `cases`
   - `case_categories`
2. 日志与统计表：
   - `view_logs`
   - `view_logs_daily`
3. 对外开放页面只返回必要字段。
4. 排行和趋势优先走聚合表，不走临时扫表。

### 4. 存储规范

1. 永久保存 `fileID`
2. 展示再换 `tempURL`
3. 富文本图片、头像、封面统一命名目录
4. 删除业务数据时，顺手评估孤儿文件清理

### 5. 富文本规范

1. 客户端编辑器只能做基础白名单
2. 服务端必须再净化一次
3. 富文本图片统一转 `cloud://` 落库
4. 渲染前再做临时链接转换
5. 对外展示前接内容安全

### 6. 日志规范

1. 不记录验证码
2. 不记录私钥 / ticket / token
3. 手机号脱敏
4. `openid` 只在必要排障场景下记录，且最好脱敏或哈希
5. 统计接口少打入参大对象日志

### 7. 发布与回滚规范

1. 每次改代码前先备份
2. 发版前确认：
   - 代码
   - 云函数
   - 控制台规则
   - 索引
   - 告警
3. 发版后留回滚点
4. 富文本、分享、文件、权限相关改动要重点回归

## 六、最容易踩坑的 30 个点

1. 把测试环境 ID 写死进正式代码。
2. 以为前端 `wx.cloud.init` 会自动决定云函数环境。
3. 把 `tempURL` 存进数据库。
4. 以为走云函数就天然安全。
5. 用简易权限长期顶替安全规则。
6. 未登录模式开了，却没给数据库 / 存储 / 云函数分别设规则。
7. 在云函数里依赖本地磁盘长期存在。
8. 忘了云函数默认 UTC+0。
9. 大量 `count()`、全表查、循环查排行。
10. 没建索引就上统计页。
11. 富文本只在前端限制，不在服务端净化。
12. 上传图片后直接存临时地址。
13. 日志里打印完整 event。
14. 把验证码、手机号、openid 直接打日志。
15. 把密钥文件提交进仓库。
16. 把开发绕过逻辑带进正式环境。
17. 依赖 `latest` 导致构建不可复现。
18. 运行时依赖外部 CDN，没有降级方案。
19. 云函数没有幂等性，重复触发就写乱数据。
20. 多表更新不考虑事务或补偿。
21. 删除分类、员工、案例后不清理关联数据。
22. 公开接口返回过多内部字段。
23. 统计日志长期不归档。
24. 定时任务没有失败告警。
25. 控制台权限默认放太大。
26. Web / 公众号场景直接照搬小程序登录态。
27. 忽略第三方 Cookie 限制。
28. 富文本 / 评论 / 用户输入不上内容安全。
29. 发版只传前端，不同步云函数和控制台配置。
30. 没有可回滚包就直接上线。

## 七、当前项目推荐技术路线

对于这个项目，后续继续开发时，推荐路线不是追求花哨，而是追求稳：

1. 小程序端
   - 负责展示、采集输入、上传文件
2. 管理端
   - 负责后台维护、富文本编辑、数据预览
3. 云函数
   - 负责鉴权、核心写操作、统计、图片地址转换、业务规则
4. 云数据库
   - 负责主数据与归档统计
5. 云存储
   - 负责头像、案例封面、富文本图片
6. 控制台
   - 负责权限、规则、索引、告警、配额

## 十、一句话总结

微信云开发真正难的，不是“把接口跑通”，而是同时把：

1. 环境
2. 权限
3. 数据
4. 文件
5. 云函数
6. 安全
7. 运维

七件事一起管稳。  
以后只要按这份知识库来做，小程序开发会明显更稳、更快，也更不容易踩坑。
