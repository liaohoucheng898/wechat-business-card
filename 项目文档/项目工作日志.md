## 四、富文本图片统一标准（2026-04-21 14:31:27）

本节只记录“富文本里的图片”这一件事。以后接手的人，如果遇到富文本图片的保存、预览、浏览、链接失效问题，以本节为唯一准则；前面时间轴里涉及图片的旧说法，不再单独采用。

### 1. 正确目标

1. 后台编辑器里，图片要能立即看到。
2. 点保存后，数据库里不能保存过期临时地址，必须保存真实云文件 ID。
3. PC 端预览时，图片要能被重新换成当前可访问地址。
4. 小程序浏览时，图片也要能被重新换成当前可访问地址。
5. 无论是新图片还是旧图片，只要节点里还保留 `data-cloud-file-id`，都要尽量恢复显示。

### 2. 正确操作方法

#### 2.1 后台编辑时

1. 用户在富文本编辑器里插入图片。
2. 后台先把图片上传到云存储。
3. 编辑器里显示的是临时可访问地址，目的是让用户当场预览。
4. 同时必须给图片节点补上真实云文件 ID：
   - `data-cloud-file-id="cloud://..."`

#### 2.2 保存到数据库时

1. 保存前，不能把临时地址直接存进数据库。
2. 正确做法是把图片节点恢复成“存储态”：
   - `src="cloud://..."`
   - `data-cloud-file-id="cloud://..."`
3. 这样数据库里保存的是稳定标识，不是会过期的地址。

#### 2.3 PC 端预览时

1. 预览页面不能直接相信图片当前 `src`。
2. 应优先按下面顺序找真实图片：
   - 先看 `data-cloud-file-id`
   - 如果没有，再看 `src` 里是不是 `cloud://...`
3. 拿到真实云文件 ID 后，再去换成最新临时地址显示。

#### 2.4 小程序浏览时

1. 小程序页面不要自己硬写图片地址转换规则。
2. 统一由云函数共用逻辑先把 HTML 处理好，再返回给页面。
3. 云函数处理时也要优先按下面顺序找真实图片：
   - 先看 `data-cloud-file-id`
   - 如果没有，再看 `src` 里是不是 `cloud://...`
4. 拿到真实云文件 ID 后，再统一换成最新临时地址。

### 3. 这次最终确认的稳定规则

1. 数据库存储的富文本图片，目标状态应是：
   - `src="cloud://..."`
   - `data-cloud-file-id="cloud://..."`

2. 后台编辑器内部允许显示临时地址，但只允许停留在“编辑态”，不能把它当作最终存储值。

3. 预览和浏览时，如果遇到旧数据是这种情况：
   - `src` 已经是旧临时地址
   - 但 `data-cloud-file-id` 还在
   也必须优先使用 `data-cloud-file-id` 重新换新地址。

4. 返回给小程序的图片 `src`，不能把地址里的 `&` 再写成 `&amp;`。
   浏览器很多时候还能显示，但小程序 `rich-text` 更容易因此断图。

### 4. 这次踩过且必须避免的坑

1. 不能把临时地址直接保存进数据库。
   这是最容易导致“刚上传能看，过一阵就看不见”的根因。

2. 不能只信任图片当前 `src`。
   很多旧数据里，真正能恢复图片的是 `data-cloud-file-id`，不是 `src`。

3. 不能只修 PC 端，不测小程序端。
   浏览器能显示，不代表小程序 `rich-text` 一定能显示。

4. 不能把云函数返回给小程序的图片地址做成 HTML 实体化链接。
   特别是带签名参数的地址，如果把 `&` 写成 `&amp;`，小程序端可能直接断图。

5. 改了共用富文本图片逻辑后，不能只改一个云函数副本。
