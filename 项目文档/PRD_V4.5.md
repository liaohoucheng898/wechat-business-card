## 一、产品概述

### 1.1 背景与目标

公司旗下有3个品牌主体，共用同一套员工团队（约20人）。员工日常需要根据业务场景，以不同公司身份向客户传递联系方式及公司信息。纸质名片成本高、更新慢、多品牌管理混乱；现有电子名片工具无法满足"一人多公司名片"的需求。

三家品牌主体为：**华悦、华宝、卓辰**。

**产品目标：** 以微信小程序为载体，让每位员工根据业务需要，自由选择以哪家公司身份分享名片给客户，客户无需下载任何 App 即可查看。同时提供 PC 端管理后台，供管理员完成所有管理操作。

### 1.2 核心设计原则

```
个人信息（姓名/手机/微信/头像）→ 每人维护一份，所有公司名片共用
职位信息 → 按公司独立维护，同一员工在不同公司可有不同职位
公司信息（名称/Logo/地址/电话/定位/介绍/案例）→ 3家公司各自独立维护，互不干扰
```

同一员工张三，可拥有最多3张名片：
- 张三 × 华悦（销售顾问）→ 分享给甲客户
- 张三 × 华宝（项目经理）→ 分享给乙客户
- 张三 × 卓辰（技术总监）→ 分享给丙客户

### 1.3 核心价值

| 角色 | 获得的价值 |
|------|-----------|
| 员工 | 一键切换公司身份分享名片，自主维护个人信息，查看各公司名片数据 |
| 客户 | 随时查看联系人信息 + 了解对应公司实力 + 一键导航到公司地址 + 一键复制手机号 |
| 管理员 | 统一维护多家公司品牌内容，管控账号，掌握全公司数据（**管理功能在 PC 端后台操作**）；**同时也是员工，拥有自己的名片** |

### 1.4 范围说明

**小程序端覆盖：**
- 员工端：登录绑定、名片预览、编辑个人信息、分享名片、个人数据统计
- 访客端：名片展示、公司介绍、业务介绍、客户案例、一键拨号/复制/导航
- 本版本为**公司内部使用的 MVP 版本**，不对外开放注册
- 公司品牌主体固定为3家（华悦、华宝、卓辰），不支持动态增减
- **需上架微信小程序商店**，需提前准备相关类目资质
- **最低支持微信基础库版本：2.25.0**（需覆盖 `wx.chooseLocation`、`wx.openLocation`、`rich-text` 组件等核心 API）
- 低于最低版本的用户，进入小程序时提示"请升级微信至最新版本以获得最佳体验"

**PC 端管理后台覆盖：**
- 管理员登录（仅账号密码）
- 员工账号管理（列表、新增、编辑、停用/启用）
- 员工密码管理（临时密码初始化 / 重置密码，用于员工首次绑定或忘记密码后重新绑定）
- 公司内容管理（3家公司基础信息 + 公司简介/业务介绍富文本编辑）
- 客户案例管理（案例列表、新增、编辑含富文本详情、删除软删除）
- 企业数据统计（全公司维度数据统计 + 趋势图 + 员工排行榜）
- 案例详情的富文本图文混排编辑（小程序端富文本编辑体验差，不适合复杂排版）
- 公司简介、业务介绍的富文本编辑（同理）

### 1.5 技术选型

**小程序端：**
- 微信小程序原生开发 + 微信云开发（云函数 + 云数据库 + 云存储）

**PC 端管理后台：**

| 项 | 选择 | 理由 |
|----|------|------|
| 前端框架 | **Vue 3 + Element Plus** | 后台管理系统生态成熟，Element Plus 表格/表单/弹窗组件开箱即用，20人规模不需要重型方案 |
| 富文本编辑器 | **TinyMCE 6** | 对 HTML 输出格式控制更精确，图片上传可自定义（对接云存储），比 Quill 更适合管理后台 |
| 地图选点 | **腾讯地图 JavaScript API** | 替代小程序端 `wx.chooseLocation`，支持搜索 + 点击选点 + 返回经纬度 |
| 图表 | **ECharts 5** | 折线图、趋势图，生态成熟 |
| 云开发调用 | **@cloudbase/js-sdk** | Web 端调用微信云开发云函数的官方 SDK |

**共用后端：**
- 两端共用同一套微信云开发资源（云函数、云数据库、云存储）

### 1.6 数据格式契约

**小程序端与 PC 端管理后台共同遵守：**

| 数据项 | 存储格式 | 说明 |
|--------|----------|------|
| 案例详情正文 | **HTML 字符串** | PC 端使用 TinyMCE 编辑器输出 HTML；小程序端使用 `rich-text` 组件渲染 |
| 公司简介 | **HTML 字符串** | PC 端编辑，小程序端 `rich-text` 渲染 |
| 业务介绍 | **HTML 字符串** | PC 端编辑，小程序端 `rich-text` 渲染 |
| 富文本中的图片 | **云存储文件 ID** | PC 端上传图片至云存储，HTML 中 `<img src>` 存储云文件 ID；小程序端渲染前批量转换为临时 URL |
| 富文本图片尺寸 | 宽度 ≤750px | 上传时自动压缩，确保小程序端渲染性能 |

### 1.7 已知限制（MVP 版本）

| 限制项 | 说明 | 后续版本计划 |
|--------|------|-------------|
| 并发编辑 | 保存时通过 `updatedAt` 时间戳比对保护（见 11.1.4） | V2 加入乐观锁 |
| 员工搜索 | 员工列表不支持搜索（20人规模暂不需要） | 人数超50后加入 |
| 小程序码 | 不支持生成名片小程序码（线下扫码场景） | V2 支持 |
| 朋友圈分享 | 暂不支持分享到朋友圈 | 视微信开放能力评估 |
| 存入通讯录字段范围 | 已支持访客在名片页点击“存入通讯录”，调用 `wx.addPhoneContact` 拉起系统通讯录保存界面；最终是否保存由用户确认 | V2 评估更多字段映射和权限提示优化 |
| 横向浏览同事名片 | 访客端不支持查看同公司其他员工名片 | V2 评估 |
| 统计排除误差 | 员工清除缓存或换设备后查看他人名片会被计入统计，基于 session 排除，MVP 阶段可接受 | V2 评估 openid 方案 |

## 二、用户角色

### 2.1 角色定义

| 角色 | 描述 | 数量 |
|------|------|------|
| **管理员** | **同时也是员工**。拥有自己的名片，可像普通员工一样使用名片功能；额外拥有管理权限（创建员工账号、维护公司内容、管控账号状态、查看全公司数据），**管理功能在 PC 端后台操作** | **2人**（互为备份）|
| **员工** | 自主维护本人名片个人信息，按需选择公司身份分享名片，查看各公司名片数据 | ~20人（含管理员）|
| **客户（访客）** | 收到分享链接后查看名片，无需登录 | 不限 |

> 管理员不是独立角色，而是"拥有管理权限的员工"。管理员在 `staff` 表中有自己的员工记录，登录后进入员工视图（我的名片）。管理功能统一在 PC 端后台操作。
> 管理员身份判定：手机号 ∈ `admin_config.adminPhones` 且 `staff.isAdmin = true`。

### 2.2 角色权限对比

| 功能 | 管理员（员工+管理权限）| 普通员工 | 客户 |
|------|----------------------|---------|------|
| 查看自己名片（各公司）| ✅ | ✅ | — |
| 分享自己名片（选择公司）| ✅ | ✅ | — |
| 编辑**本人**名片个人信息 | ✅ | ✅ | ❌ |
| 编辑**他人**名片信息 | ✅ 在 PC 端后台操作 | ❌ | ❌ |
| 新增 / 停用员工账号 | ✅ 在 PC 端后台操作 | ❌ | ❌ |
| 配置员工开通哪些公司名片及对应职位 | ✅ 在 PC 端后台操作 | ❌ | ❌ |
| 编辑公司内容（3家）| ✅ 在 PC 端后台操作 | ❌ | ❌ |
| 管理客户案例（3家）| ✅ 在 PC 端后台操作 | ❌ | ❌ |
| 查看本人数据统计（各公司）| ✅ | ✅ | ❌ |
| 查看全公司数据统计 | ✅ 在 PC 端后台操作 | ❌ | ❌ |
| 查看他人名片 | ✅ | ✅ | ✅（需链接）|

## 三、核心用户旅程

> 每条旅程标注了**关键页面**和**可能出错的节点**，开发和测试时重点关注 ⚠️ 标记处。

### 3.1 访客旅程

**旅程 C：客户查看名片（核心价值交付）**

```
客户在微信聊天中点击分享卡片
  → 进入名片展示页 → 看到员工姓名、职位、公司名、头像
  → ⚠️ 触发 card_view 统计日志写入
  → [可选] 点击手机号 → 弹出操作菜单 → 拨打电话 / 复制号码
  → [可选] 点击微信号旁"复制"按钮 → 复制到剪贴板
  → [可选] 点击"存入通讯录" → 拉起系统通讯录保存界面
  → [可选] 查看"公司介绍"内容块 → 点击地址打开地图 / 无坐标时复制地址
  → [可选] 查看"业务介绍"内容块 → 阅读公司业务内容
  → [可选] 查看"客户案例"内容块 → 点击案例计入统计
  → 关闭
```

**旅程 D：客户打开已停用员工的名片**

```
客户点击之前保存的分享链接
  → ⚠️ 系统检测 staff.status = disabled
  → 展示"名片已注销"页 → 显示对应公司名称和电话 → 客户可点击拨打公司电话
```

### 3.2 员工旅程

**旅程 A：新员工首次使用（从0到拥有名片）**

```
前置条件：管理员已在 PC 端后台创建该员工账号（姓名+手机号+开通公司+职位）

前置条件补充：管理员创建员工后，系统自动生成 8 位临时密码，并由管理员通知员工“账号=手机号，密码=临时密码”

员工直接点击小程序图标打开
  → 微信环境启动 → 系统优先检测本地 session / 当前微信是否已绑定
  → 已绑定当前微信或本地 session 有效 → 默认进入"我的名片"页
  → ⚠️ 未绑定且无有效 session → 默认进入"账号密码登录"页
  → 输入账号（手机号）+ 临时密码 → 登录成功
  → ⚠️ 系统检测当前微信尚未加入该员工的绑定列表 → 自动追加绑定当前微信
  → 写入 session
  → 进入"我的名片"页 → 看到管理员预设的公司名片（可能头像是默认的）
  → 点击"编辑名片" → 上传头像、补充微信号等 → 保存
  → 点击"分享名片" → 选择公司 → 分享给客户
```

**旅程 B：员工日常分享名片（高频场景）**

```
员工后续再次点击小程序图标打开 → 自动登录（当前微信命中员工绑定列表）→ 直接进入"我的名片"页
  → 如有多家公司，顶部 Tab 切换到目标公司预览
  → 点击"分享名片" → 选择公司（若只有1家则跳过选择）→ 微信分享给客户
  → 完成，≤3步操作
```

**旅程 E：员工忘记密码（低频但管理员协同）**

```
员工联系管理员反馈忘记密码
  → 管理员进入 PC 后台"员工管理" → 点击"重置密码"
  → 系统生成新的临时密码 → 管理员把密码发给员工
  → 员工重新打开小程序 → 输入账号（手机号）+ 新密码
  → 绑定成功 → 进入"我的名片"页
```

### 3.3 管理员旅程

**旅程 A：系统初始化（上线第一天）**

```
管理员在电脑上打开后台地址
  → 默认进入"账号密码登录"
  → 输入账号（手机号）+ 密码 → 登录
  → ⚠️ 若当前为临时密码 → 先强制修改密码
  → 进入案例管理页
  → 通过左侧菜单进入员工管理页（空列表）
  → 逐个新增员工：填写姓名、手机号 → 勾选开通的公司 → 填写各公司职位 → 保存
  → 系统为每个员工自动生成 8 位临时密码
  → ⚠️ 所有员工创建完成后，通知员工使用“手机号 + 临时密码”登录小程序
  → 进入公司管理 → 逐个编辑3家公司信息：
      上传 Logo → 用腾讯地图选点填入地址/经纬度 → 用 TinyMCE 编辑公司简介和业务介绍 → 保存
  → 进入案例管理 → 新增案例（选公司、填标题、上传封面、编辑富文本详情）→ 保存
  → 完成，员工可以开始使用名片了
```

**旅程 B：日常管理 — 新增员工（单个录入）**

```
管理员登录后台 → 员工管理 → 点击"新增员工"
  → 填写新员工信息 → 勾选开通公司 → 填写职位 → 保存
  → 系统自动生成临时密码并弹窗展示一次
  → ⚠️ 告知新员工：账号=手机号，密码=临时密码 → 新员工打开小程序登录并完成自动绑定
```

**旅程 B1：日常管理 — 批量导入员工**

```
管理员登录后台 → 员工管理 → 点击"导入员工"
  → 下载Excel模板
  → 填写姓名、手机号、华悦职位、华宝职位
  → 重新上传Excel → 点击"开始导入"
  → 系统按职位自动开通对应公司：
     · 填了华悦职位 → 自动开通华悦
     · 填了华宝职位 → 自动开通华宝
  → 弹出导入结果：总记录数、成功数、失败数、失败明细
```

### 4A.2 小程序端 Session 状态机

```
              ┌─────────────────┐
              │  无 Session      │ ←── token 过期且静默登录失败 / 被停用清除
              └────────┬────────┘
                       │ 绑定成功 / openid 静默登录 → 生成 32位 token，7天有效
                       ↓
              ┌─────────────────┐
              │  有效 Session    │
              └────────┬────────┘
                       │ token 进入到期前 5 分钟 → 前端清理本地 session
                       │ 下次启动 / 重新进入时 ↓
              ┌─────────────────┐
              │  已过期          │ → 当前微信命中绑定列表 → 静默重登 → 回到“有效”
              └─────────────────┘
```

### 4A.3 PC 端管理员 Session 状态机

```
              ┌─────────────────┐
              │  未登录          │ ←── 主动退出 / SDK 登录态丢失 / 路由守卫校验失败
              └────────┬────────┘
                       │ adminLogin → 返回 ticket
                       │ signInWithTicket(ticket) → 建立 CloudBase 登录态
                       ↓
              ┌─────────────────┐
              │  已登录          │ → SDK 登录态失效 / 手动退出 → 回到“未登录”
              └─────────────────┘
```

**两端 Session 对比：**

| 对比项 | 小程序端 | PC 端 |
|--------|---------|-------|
| 凭证字段 | `sessionToken` | CloudBase SDK 登录态（业务侧仅缓存 `adminInfo`） |
| 有效期 | 7天 | 由 CloudBase 登录态控制，业务层未额外约定固定时长 |
| 续期 | 不做服务端自动续期，过期后走 openid 静默重登 | 由 SDK 自身机制维护 |
| 过期处理 | 前端清理本地 session，重新走静默登录 | 路由守卫发现登录态无效后跳转登录页 |
| 互相影响 | ❌ 完全独立 | ❌ 完全独立 |

- **P2：** 增值功能，可在后续版本迭代

### 5.2 访客端 — 名片展示页

> 客户点开员工分享的链接后看到的页面，**无需登录**。
> 链接中包含员工 ID 和公司 ID，页面内容根据参数动态渲染。

#### 5.2.1 页面加载与异常处理 【P0】

**正常加载流程：**
1. 解析 URL 参数 `staffId` 和 `companyId`
2. 查询 `staff` 表获取员工信息，校验 `status`
3. 查询 `companies` 表获取公司信息
4. 校验该员工是否开通了该公司名片（`enabledCompanies` 中是否包含该 `companyId`）
5. 全部通过 → 渲染名片页
6. 触发统计日志写入（见 5.2.5）

**异常状态矩阵：**

| 场景 | 展示内容 |
|------|----------|
| `staffId` 参数缺失或非法 | 通用错误页："名片不存在" |
| `companyId` 参数缺失或非法 | 通用错误页："名片不存在" |
| `staffId` 在 `staff` 表中不存在 | 通用错误页："名片不存在" |
| `companyId` 在 `companies` 表中不存在 | 通用错误页："名片不存在" |
| 员工存在但未开通该公司名片 | 通用错误页："名片不存在" |
| 员工状态为 `disabled` | 名片已注销页（见 5.2.4） |
| 网络异常 / 云函数超时 | "加载失败，请点击重试" + 重试按钮 |

#### 5.2.2 个人信息区 【P0】

**内容块 1 — 公司介绍**

#### 5.2.5 访问统计触发规则 【P0】

| 触发动作 | 写入日志类型 | 备注 |
|----------|------------|------|
| 进入名片展示页 | `card_view` | **内部员工访问不计入**（按已绑定微信识别）；同一外部访客10分钟内重复打开同一员工同一公司名片不重复记一次 |
| 切换至公司介绍 Tab | `company_intro_click` | 每次切换记录1次 |
| 点击某条案例进入详情 | `case_click` | 记录具体案例 ID |

**排除逻辑实现方式：**

名片展示页加载时，云端按当前微信 `OPENID` 判断是否记统计：

```
云函数读取当前微信 OPENID
    → OPENID 已绑定到任一员工账号 → 不写入日志
    → OPENID 未绑定内部员工 → 继续进入外部访客统计
         → 同一 OPENID + staffId + companyId + logType 10分钟内已记录过 → 不重复写入
         → 未命中防重 → 写入日志
```

> **设计决策说明：** 排除逻辑基于云函数 `OPENID` 与员工绑定关系判断，而非前端本地 session。原因：
> 1. 微信云函数可直接获取当前调用者 `OPENID`，不依赖本地 session
> 2. 员工清除小程序缓存或更换设备后，只要仍使用已绑定微信打开名片，仍不会计入访客统计
> 3. 统计继续只保留外部访客流量，避免内部流量混入浏览量
> 4. 管理员同样通过 session 排除，无需额外逻辑

> `company_intro_click` 和 `case_click` 日志在 MVP 统计页面中**不展示**，仅做数据储备，V2 版本在统计页面增加展示。

**日志写入字段：**

| 字段 | 说明 |
|------|------|
| `staffId` | 被查看的员工 ID |
| `companyId` | 被查看的公司 ID |
| `logType` | 日志类型（`card_view` / `company_intro_click` / `case_click`） |
| `caseId` | 案例 ID（仅 `case_click` 类型有值，其余为 null） |
| `shareScene` | 分享来源场景（见 5.3.4） |
| `createdAt` | 日志创建时间戳 |

### 5.3 员工端

#### 5.3.1 登录与绑定机制 【P0】

**登录方式：**
- 默认：账号密码绑定（账号 = 手机号）
- 已绑定当前微信且 session 仍有效时，可自动登录，无需重复输入账号密码
- 已绑定员工直接点击小程序图标时，默认进入“我的名片”页；未绑定员工或陌生人直接点击小程序图标时，默认进入账号密码绑定页
- 通过员工分享链接或客户分享卡片打开时，仍优先进入访客名片页，不参与“我的名片”默认打开分流
- 同一员工支持绑定多个微信号，不限制数量；多个已绑定微信号共用同一张名片、同一份资料和同一份统计

**账号规则：**

| 项目 | 规则 |
|------|------|
| 登录账号 | 员工手机号 |
| 初始密码 | 系统自动生成 8 位临时密码 |
| 密码用途 | 首次绑定当前微信；忘记密码后由管理员重置并重新绑定 |
| 重置密码 | 管理员可重置，系统重新生成 8 位临时密码 |
| 默认入口 | 账号密码绑定 |
| 备用入口 | 无 |
| 微信绑定数量 | 不限制，支持同一员工绑定多个微信号 |

**流程 A：已绑定微信自动登录**

```
员工直接点击小程序图标打开小程序
    → 系统检测本地 session 是否有效
        → 有效 → 直接进入"我的名片"页
        → 无效 → 检测当前微信是否已存在于员工绑定列表
            → 已绑定且状态 active → 下发新 session → 进入"我的名片"页
            → 已绑定但状态 disabled → 提示"账号已停用，请联系管理员"
            → 未绑定 → 进入默认登录页
```

**流程 B：默认账号密码登录**

```
员工首次打开小程序 / 自动登录失败
    → 进入"账号密码绑定"页（默认）
    → 输入账号（手机号）+ 密码
    → 系统用手机号匹配 staff 表中管理员预创建的账号
        → 未匹配到账号 → "该手机号尚未开通名片，请联系管理员"
        → 匹配到但账号状态 disabled → "账号已停用，请联系管理员"
        → 密码错误 → 计入错误次数
        → 连续错误5次 → 锁定15分钟
        → 密码正确 → 继续下一步
    → 检测当前微信绑定关系
        → 当前微信尚未加入该员工绑定列表 → 自动追加绑定到该员工账号
        → 当前微信已在该员工绑定列表中 → 直接通过
    → 写入本地 session
    → 进入"我的名片"页
```

**密码安全策略：**

| 策略 | 规则 |
|------|------|
| 临时密码长度 | 8 位 |
| 临时密码生成方式 | 系统随机生成，创建员工和重置密码时触发 |
| 连续错误锁定 | 连续输错 5 次后锁定 15 分钟 |
| 重置后处理 | 旧密码立即失效；员工需使用新临时密码重新完成账号绑定 |

**异常处理：**

| 场景 | 提示文案 |
|------|---------|
| 手机号未匹配到任何账号 | "该手机号尚未开通名片，请联系管理员" |
| 账号已停用 | "账号已停用，请联系管理员" |
| 密码错误 | "账号或密码错误，请重新输入" |
| 连续错误锁定 | "密码错误次数过多，请15分钟后重试" |
| 网络异常 | "网络连接失败，请检查网络后重试" |
| 第二个或更多微信号绑定 | 输入同一员工账号密码后，自动加入该员工绑定列表 |

**Session 管理策略：**

| 策略 | 规则 |
|------|------|
| Token 生成方式 | 云函数生成 32 位随机字符串，账号密码绑定成功或静默登录成功时下发 |
| 存储内容 | 本地缓存 `staffId` + `token` + `expireAt` |
| 有效期 | 30 天 |
| 续期机制 | 每次打开小程序时，若 token 有效且剩余有效期 ≤7 天，自动续期 30 天 |
| 过期处理 | token 过期 → 先尝试通过当前微信 openid 静默重新登录；失败则回到账号密码登录页 |
| 校验方式 | 员工端所有云函数入口校验 token 有效性（比对数据库存储的 token + 过期时间）|
| Token 存储位置 | 服务端在 `staff` 表增加 `sessionToken` 和 `sessionExpireAt` 字段 |
| 强制失效 | 员工被停用或管理员重置密码时，下次请求返回失效状态 → 前端清除本地 session |

> 访客端不涉及 token，仅员工端使用。

#### 5.3.2 我的名片页（员工首页）【P0】

**页面结构：单页面 + 底部固定按钮，无 TabBar。**

**所有员工**打开小程序 → 自动登录 → 直接进入"我的名片"页。

- 若员工开通了**多家公司名片**，页面顶部显示公司切换 Tab（如：华悦 / 华宝 / 卓辰）
- 若员工只开通了**1家公司名片**，不显示 Tab，直接展示该公司名片
- 切换 Tab 后，名片预览内容（公司名、Logo、地址、介绍、案例、职位）随之切换
- 个人信息区中，**姓名、手机、微信、邮箱、头像、个人简介** 各公司名片共用，不随 Tab 切换
- **职位** 随 Tab 切换，展示该员工在当前所选公司下的职位

**空状态：** 员工已登录但 `enabledCompanies` 为空时（数据异常场景），展示引导页："您的名片尚未配置，请联系管理员开通"。仅保留"编辑名片"按钮，隐藏"分享名片"和"数据统计"。

**底部固定操作按钮：**

| 按钮 | 动作 | 可见性 |
|------|------|--------|
| 编辑名片 | 进入个人信息编辑页 | 所有员工 |
| 分享名片 | 触发分享流程（见 5.3.4）| 所有员工（`enabledCompanies` 非空时） |
| 数据统计 | 进入个人数据统计页 | 所有员工（`enabledCompanies` 非空时） |

#### 5.3.3 编辑本人名片 【P0】

员工可自主修改以下**个人信息字段**（所有公司名片共用）：

| 字段 | 是否必填 | 校验规则 |
|------|----------|---------|
| 头像 | 必填 | 支持相册选择或拍照上传；格式限 JPG/PNG；原图 ≤5MB；当前保存为员工统一头像文件，PC 端员工管理与小程序编辑名片共用同一张头像 |
| 姓名 | 必填 | 1-20个字符；不允许纯空格；不允许特殊符号（仅中英文、数字、·） |
| 微信号 | 选填 | 6-20个字符；字母开头；仅允许字母、数字、下划线、减号 |
| 邮箱 | 选填 | 标准邮箱格式：`/^[^\s@]+@[^\s@]+\.[^\s@]+$/` |
| 个人简介 | 选填 | ≤100字，超出实时提示剩余字数，达到上限后禁止继续输入 |

> **职位字段员工不可自行编辑**（因职位按公司区分，由管理员在 PC 端后台员工账号管理中配置）。
> **公司相关信息员工不可编辑**，由管理员在 PC 端后台统一维护。
> **手机号仅展示，不允许员工在小程序内自行修改。**

**图片上传交互与同步规则：**

```
员工点击头像区域
    → 弹出选择：相册 / 拍照
    → 选择图片后 → 进入1:1裁剪界面
    → 裁剪完成 → 显示上传 loading
    → 上传至云存储
    → 保存该头像文件ID到员工资料
    → PC 端列表、小程序员工页、客户分享页统一基于同一员工头像数据读取
    → 页面展示时，由接口把云文件ID转换为临时可访问图片地址
    → 上传失败（网络异常）→ Toast "头像上传失败，请重试" → 恢复原头像显示
```

**交互流程：**
1. 进入编辑页，各字段回显当前数据；**同时记录当前 `updatedAt` 值，用于并发保护校验（见 11.1.4）**
2. 修改后点击"保存"
3. 前端校验 → 通过后调用云函数写入云数据库（含并发保护比对）
4. 该员工所有公司名片的个人信息区**实时生效**
5. 保存成功 → Toast "保存成功" → 自动返回"我的名片"预览页
6. 保存失败 → Toast "保存失败，请重试"

#### 5.3.4 分享名片 【P0】

**开通1家公司的员工：**
- 点击"分享名片" → 直接调起微信原生分享，无需选择

**开通多家公司的员工：**
- 点击"分享名片" → 弹出选择框（仅展示该员工已开通的公司）→ 选择后调起微信原生分享

**分享卡片内容：**

| 元素 | 内容 |
|------|------|
| 标题 | `[姓名] · [职位] · [所选公司名]` |
| 描述 | `有幸相识，愿您乘风破浪` |
| 封面图 | 使用微信默认分享效果，不单独维护分享封面图字段 |
| 跳转路径 | `/pages/card/index?staffId=[员工ID]&companyId=[公司ID]&scene=forward` |

**说明：**
- 小程序端与名片打开后二次转发，统一走微信默认分享效果
- 产品侧不再维护 `shareCover` 字段，也不再提供分享封面图上传入口

**分享场景记录：**

> ⚠️ `wx.shareAppMessage` 的回调不返回分享目标类型（好友/群），微信已于2018年取消此能力。

**方案：** 通过**被打开时的场景值**反向推断分享来源：

| 分享方式 | 触发 API | 打开时 scene 值 | `shareScene` 记录值 |
|----------|---------|----------------|-------------------|

### 6.1 登录页

**登录方式：** 账号（手机号） + 密码

**默认账号密码登录流程：**

```
管理员打开后台页面
  → 默认看到"账号密码登录"
  → 输入账号（手机号）+ 密码
  → 调用 adminPasswordLogin（规划新增）：
      1. 校验手机号格式
      2. 校验手机号 ∈ admin_config.adminPhones
      3. 查询 staff 表，校验 isAdmin = true 且 status = active
      4. 校验密码是否正确
      5. 若连续输错5次 → 锁定15分钟
      → 全部通过 → 返回 staffInfo + ticket + passwordStatus
      → 手机号不在管理员列表 → "无管理权限"
      → 账号已停用 → "账号已停用"
      → 密码错误 → "账号或密码错误"
      → 临时密码首次登录 → 登录成功后强制进入"修改密码"
  → 前端调用 `signInWithTicket(ticket)` 建立 CloudBase 登录态
  → 缓存 `adminInfo`
  → 若为临时密码 → 跳转"修改密码"页
  → 否则跳转首页（案例管理页）
```

**Session 管理：**

| 策略 | 规则 |
|------|------|
| 登录凭证 | CloudBase Custom Login ticket 只用于换取 SDK 登录态，业务层不再单独保存 `pcToken` |
| 过期处理 | 路由守卫调用 `auth.getLoginState()` 检查；失效后自动跳转登录页 |
| 主动退出 | 顶部导航栏"退出登录"按钮，调用 CloudBase `signOut()` 并清除 `localStorage.adminInfo` |
| 临时密码处理 | 使用临时密码登录成功后，必须先修改密码，修改前不可进入后台业务页 |

**密码安全策略：**

| 项目 | 规则 |
|------|------|
| 账号 | 员工手机号 |
| 初始密码 | 系统自动生成 8 位临时密码 |
| 新密码规则 | 8-20 位，必须包含字母和数字 |
| 连续错误 | 连续输错 5 次锁定 15 分钟 |
| 日志要求 | 不记录明文密码，不记录完整密文 |
| 重置后处理 | 旧密码立即失效，新临时密码首次登录后强制修改 |

**入口：** 员工列表操作列点击"删除"

**交互流程：**
- 点击"删除" → 二次确认弹窗："确定删除该员工？删除后该员工账号将无法恢复使用。"
- 确认 → 调用 `adminDeleteStaff`

  - 不能删除最后一个管理员

|--------|--------|--------|

| 微信绑定关系（openid / openids） | 保留（不解除绑定） | 自动恢复，无需重新绑定 |
| 本地 session | 下次打开时后端校验 status → 返回"账号已停用" → 清除本地 session | 员工重新打开 → 当前微信命中绑定列表 → 正常登录 |
| 管理员身份 | isAdmin 不变，但因无法登录而无法使用管理功能 | 恢复管理功能 |

**管理员变更方案：**

提供 `transferAdmin` 云函数脚本，确保原子性操作：
- `action = 'add'`：校验 targetPhone 在 staff 表中存在 → 加入 adminPhones → 若该员工已有已绑定微信，则把所有已绑定微信加入 adminOpenids → 设 isAdmin=true → 记录日志
- `action = 'remove'`：校验至少保留1个管理员 → 从 adminPhones 移除 → 从 adminOpenids 移除该员工所有已绑定微信 → 设 isAdmin=false → 记录日志
- 调用方式：MVP 阶段通过云开发控制台手动调用

#### 11.1.3 网络异常处理

- 所有云函数调用统一封装 try-catch
- 超时时间：10秒
- 超时或网络错误 → 展示"加载失败，请点击重试" + 重试按钮
- 重试次数上限：3次，超过后提示"请检查网络连接后重新打开"

#### 11.1.4 并发编辑保护

**适用范围：** 所有涉及 `updatedAt` 字段的编辑操作（包括管理员编辑内容 和 员工编辑本人个人信息）

```
1. 进入编辑页时，获取当前记录的 updatedAt，存入页面变量 pageLoadedAt
2. 保存时，云函数比对数据库中 updatedAt 与 pageLoadedAt：
    → 相等 → 无冲突 → 正常保存 → 更新 updatedAt
    → 数据库 updatedAt 更新 → 有冲突 → 返回冲突错误
3. 前端收到冲突错误 → 弹窗："该内容已被其他管理员修改，请刷新后重新编辑"
    [刷新] → 重新加载最新数据
    [取消] → 留在当前页面
```

#### 11.1.5 表单未保存退出保护

编辑页面修改了任意字段后，用户点击返回 → 弹窗："您有未保存的修改，确定离开吗？" [离开] [继续编辑]

## 十四、微信审核与上架注意事项

> 这些不是"以后再说"的事——**开发前就要准备**，否则做完了提交审核被拒，白等一周。

### 14.1 审核避坑指南

| 常见拒绝原因 | 应对措施 |
|-------------|----------|
| 功能不完整（页面空白/按钮无响应） | 提审前必须走完所有用户旅程，空状态页面也要有内容 |
| 需登录才能体验核心功能 | 访客端不需要登录（✅ 已满足）；提审时提供测试账号给审核员 |
| 缺少隐私协议 | 在小程序设置中配置隐私协议，说明收集手机号、位置信息、通讯录写入等用途 |
| 诱导分享 | 分享按钮文案不能出现"分享有奖""转发领红包"等（本产品不涉及） |
| 类目不匹配 | 确保选择的类目与实际功能一致 |

### 14.2 隐私协议必备内容

小程序需声明以下信息收集项：

| 收集的信息 | 用途 | 使用的接口 |
|-----------|------|-----------|
| 微信 openid | 员工登录绑定 | `wx.cloud.callFunction` |
| 手机号 | 员工身份验证、名片展示 | 账号密码 |
