2. 已接通的发布范围
3. 当前项目可直接使用的发布命令
4. 发布前检查

## 二-A、CloudBase CLI 登录凭据规则

1. 以后本项目的 CloudBase 操作默认使用项目专用腾讯云子账号 API 密钥登录态，不再使用主账号扫码授权。
2. 子账号密钥只保存在本机 CloudBase CLI 登录态中，不写入仓库，不写入项目文档，不发送到聊天记录。
3. 当前项目统一通过 `scripts/invoke-tcb.cmd` 调用 CloudBase CLI，该脚本会把登录态写入项目本地 `.cloudbase-home/` 目录；该目录已被 `.gitignore` 排除，禁止提交。
4. 验证登录态是否有效时，优先执行：
   ```bash
   .\scripts\invoke-tcb.cmd env list --json
   ```
   正常情况下应返回环境 `cloud1-d1gvh2zc3c5919349`，且状态为 `NORMAL`。
5. 如果登录态失效，优先使用子账号 API 密钥重新登录：
   ```bash
   .\scripts\invoke-tcb.cmd logout
   .\scripts\invoke-tcb.cmd login --apiKeyId <SecretId> --apiKey <SecretKey>
   .\scripts\invoke-tcb.cmd env list --json
   ```
6. 除非用户明确要求或子账号权限无法满足当前排障需要，否则不要重新发起主账号扫码授权。

## 三、当前已接通的自动部署能力

当前项目已经接通以下自动部署能力：

1. 云函数发布
2. PC 后台静态网页发布
3. CloudBase 静态托管资源发布

也就是说：

3. 后台登录相关的 HTTP 服务也不需要再手动在控制台里一个个配置
4. 如果改动包含小程序页面，仍需单独走微信开发者工具流程

## 四、当前项目相关配置文件

当前项目已新增并启用以下文件：

1. `cloudbaserc.json`
   - 作用：固定当前 CloudBase 环境 ID、函数根目录和函数部署配置
   - 当前要求：必须覆盖本地真实存在的全部云函数目录；当前已对齐为 35 个业务云函数目录
2. `package.json`
   - 作用：提供统一发布命令入口
3. `scripts/invoke-tcb.cmd`
   - 作用：统一调用 CloudBase CLI，并把登录态和配置写到项目本地目录
4. `scripts/deploy-cloudfunctions.ps1`
   - 作用：批量发布 `miniprogram/cloudfunctions` 下的云函数
5. `scripts/deploy-http-routes.ps1`
   - 作用：发布后台登录相关的 CloudBase HTTP 服务
   - 当前服务包括：
     - `/auth/admin/login`
     - `/auth/admin/send-sms`
6. `scripts/deploy-admin-hosting.ps1`
   - 作用：发布 `admin/dist` 到 CloudBase 静态托管
7. `admin/.env.development`
   - 作用：后台开发环境的 CloudBase HTTP 服务基础地址
8. `admin/.env.production`
   - 作用：后台生产构建时使用的 CloudBase HTTP 服务基础地址
9. `admin/src/config/env.js`
   - 作用：后台运行时环境配置，包含 `VITE_CLOUD_HTTP_BASE_URL`
10. `.gitignore`
   - 已排除 `.cloudbase-home/`，防止本地登录态误提交到仓库

## 五、当前可直接使用的命令

以后如果需要执行 CloudBase 自动部署，统一使用项目根目录下的命令。

### 1. 查看环境

```bash
npm run cb:env:list
```

作用：
1. 查看当前账号可访问的 CloudBase 环境
2. 验证本机登录态是否有效

### 2. 重新登录 CloudBase

```bash
npm run cb:login
```

作用：
1. 当本机登录态失效时，优先使用项目专用腾讯云子账号 API 密钥重新登录
2. 登录成功后，后续可继续发布云函数、HTTP 服务和后台
3. 不再默认使用主账号扫码授权；只有用户明确要求或子账号权限不足时，才临时走主账号扫码授权排障

### 3. 只发布云函数

```bash
npm run cb:deploy:functions
```

作用：
1. 批量发布 `miniprogram/cloudfunctions` 下的云函数
2. 适用于只改了云函数的场景

### 4. 只发布 HTTP 服务

```bash
npm run cb:deploy:http-routes
```

作用：
1. 发布后台登录相关的 CloudBase HTTP 服务
2. 当前包括：
   - `/auth/admin/login`
   - `/auth/admin/send-sms`
3. 适用于改了后台登录入口、验证码入口、HTTP 服务路径映射的场景

### 5. 只构建后台

```bash
npm run cb:build:admin
```

作用：
1. 生成最新的 `admin/dist`
2. 适用于发布前的构建验证

### 6. 只发布后台静态网页

```bash
npm run cb:deploy:admin
```

作用：
1. 把 `admin/dist` 发布到 CloudBase 静态托管
2. 适用于只改了后台前端页面的场景

### 7. 一次性发布云函数、HTTP 服务和后台

```bash
npm run cb:deploy
```

作用：
1. 先发布云函数
2. 再发布后台登录相关的 HTTP 服务
3. 再构建后台
4. 最后发布后台静态托管

适用场景：
1. 本次改动同时涉及云函数和后台
2. 本次改动涉及后台登录、验证码、HTTP 服务映射
3. 想一次把后端能力和后台页面都发上去

## 六、默认发布顺序

如果本次改动同时涉及后端和前端，默认顺序为：

1. 先发布云函数或后端能力
2. 如涉及后台登录入口或验证码入口，再发布 CloudBase HTTP 服务
3. 再构建后台前端
4. 再发布 PC 后台静态网页
5. 最后验证页面、接口、登录、数据是否正常

如果只改后台页面，只发布后台。  
如果只改云函数，只发布云函数。  
如果只改 HTTP 服务映射，只发布 HTTP 服务。  
如果涉及小程序页面，仍需单独走微信开发者工具流程。

## 七、发布前检查

每次发布前，必须依次检查：

1. 当前环境 ID 是否正确
2. 本地代码是否已经是最新版本
3. 是否已经创建备份或可回滚点
4. 本次准备发布的内容范围是否明确：
   - 是云函数
   - 是 HTTP 服务
   - 是后台前端
   - 还是几者同时都有
5. 如涉及云函数发布，先确认 `cloudbaserc.json` 中列出的函数与本地真实目录一致，避免发布时落回临时默认配置
6. 如涉及敏感配置，确认未误提交到仓库
7. 如涉及后台发布，确认后台构建通过
8. 如涉及后台登录链路，确认 `VITE_CLOUD_HTTP_BASE_URL` 指向当前 CloudBase 实际可用域名

## 八、发布后验证

每次发布后，至少验证以下内容：

1. 对应云函数是否已成功更新
2. 对应 HTTP 服务是否已成功创建或更新
3. 后台页面是否已打开到最新版本
4. 相关接口是否返回正常
5. 本次修复的问题是否消失
6. 是否出现新的连带问题
7. 如静态资源存在缓存，确认是否为缓存导致结果未更新

推荐的命令级验证方式：

1. 云函数发布后，用 `tcb fn detail <函数名>` 核对函数更新时间、超时配置和云端代码关键片段
2. 后台静态托管发布后，用 `tcb hosting list --env-id <环境ID> --json` 核对本次构建产物文件名是否已出现在云端
3. 如果本地构建产物文件名与云端清单一致，再结合页面实测，避免误把旧缓存当成新版本

如本次改动涉及后台登录链路，还要额外验证：

1. 手机号 + 密码登录是否正常
2. 手机号 + 验证码登录是否正常
3. 发送验证码接口是否正常
4. CloudBase 用户管理里是否还在继续新增匿名空白用户

## 九、当前使用方式说明

从现在开始，本项目的推荐工作方式是：

1. 用户提需求
2. Codex 改代码
3. 如果本次改动需要上线，由 Codex 直接执行 CloudBase 自动部署
4. 用户只负责测试结果

也就是说：

1. 用户不需要再手动上传云函数
2. 用户不需要再手动上传后台 `dist`
3. 用户不需要再手动在 CloudBase 控制台里维护后台登录用的 HTTP 服务
4. 用户只需要在 Codex 提示后测试对应页面或功能

## 十、后台登录链路特别说明

当前 PC 后台登录链路已经从：

`先匿名登录 CloudBase -> 调云函数 -> 拿 ticket -> 正式登录`

切换为：

`直接请求 CloudBase HTTP 服务 -> 后端校验 -> 返回 ticket -> 正式登录`

当前对应的 HTTP 服务为：

1. `/auth/admin/login`
2. `/auth/admin/send-sms`

说明：

1. 这两个 HTTP 服务底层仍分别复用 `adminLogin`、`sendSmsCode` 的业务校验逻辑
2. 小程序现有登录链路没有被推翻，仍保留原云函数调用方式
3. 现在 PC 后台源码已经不再依赖“允许匿名登录”开关才能完成登录
4. 如果后续准备正式关闭 CloudBase 的“允许匿名登录”，仍应先做一次完整回归验证

## 十一、回滚方式

优先使用以下方式回滚：

1. Git 回滚到上一个可用提交
2. 使用本地备份包恢复
3. 重新发布上一个稳定版本的云函数、HTTP 服务或前端构建产物

每次发布前，都要保证至少存在一种可回滚方式。

## 十二、常见风险与避免方式

1. 不要在未确认环境 ID 的情况下执行 CloudBase 发布操作。
2. 不要凭印象编写 CLI 命令，先看现有脚本或官方帮助。
3. 不要把 AI 模板目录、兼容目录、镜像规则文件当成业务主源。
4. 不要在未确认目标范围时执行删除、覆盖、批量更新操作。
5. 不要在问题未确认解决前提前修改 PRD 或技术方案文档。
6. 向 JSON 或配置文件写入工具结果时，先转成字符串，避免写入失败。
7. CloudBase 登录态如果失效，优先按“二-A、CloudBase CLI 登录凭据规则”使用项目专用子账号 API 密钥恢复，不要默认重新发起主账号扫码授权。
8. 后台登录相关改动不能只发云函数或只发后台页面；如链路两端都变了，必须把云函数、HTTP 服务、后台一起发。
9. 如果后台登录改造后准备关闭 CloudBase 匿名登录，先验证两种登录方式都正常，再去控制台关闭。
10. 不要只看本地代码判断“已经生效”；云函数要看 `fn detail`，后台静态资源要看 `hosting list`。
11. `cloudbaserc.json` 如果缺少真实存在的云函数目录，后续发布容易出现“脚本能跑、配置不一致”的隐患，修正后再发布。
