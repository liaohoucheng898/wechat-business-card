## 一、总体结论

当前源码 **不建议直接按“安全已达标”状态继续长期上线运行**。

核心原因不是功能跑不起来，而是存在 2 个高危问题：

1. 仓库中直接提交了云开发自定义登录私钥。
2. 短信验证码流程里存在常开的开发绕过逻辑。

这两个问题都属于“不是小毛病，而是需要优先处理”的类型。  
如果不先处理，后续无论 UI 做得多好，系统的账号安全边界都不稳。

下面这些地方做得是对的，后续可以保留：

1. 管理端权限不是只看前端，而是云函数中再次校验管理员身份，并且同时校验 `staff.isAdmin` 与 `admin_config.adminPhones`。
2. 小程序端编辑资料、PC 端编辑员工/公司/案例，已经做了 `pageLoadedAt` 并发保护，能防止“后保存覆盖先保存”的脏写问题。
3. 员工停用时会清空登录态字段，方向正确。
4. 富文本中的云存储图片已经做了 `cloud://` 到临时链接的转换，解决了小程序和预览端图片无法显示的问题。
5. 管理端构建可以通过，说明当前前端代码至少没有明显的编译级错误。
6. 云函数入口文件的语法检查通过，说明当前云函数代码没有明显的 JS 语法错误。

## 三、问题清单

### F-01 `P0` 仓库中直接包含 CloudBase Custom Login 私钥

- 风险等级：严重 / 最高优先级
- 位置：
  - `miniprogram/cloudfunctions/adminLogin/tcb_custom_login.json:1`
  - `miniprogram/cloudfunctions/adminLogin/index.js:13`
- 现象：
  - `adminLogin` 云函数直接 `require('./tcb_custom_login.json')`
  - `tcb_custom_login.json` 中包含完整 `private_key`
- 风险说明：
  - 这是管理端自定义登录签票用的私钥。
  - 一旦仓库泄露、备份包外流、开发机被拷走，攻击者就有机会伪造登录票据。
  - 这类问题不是“可能不安全”，而是“密钥治理已经失守”。
- 建议：
  1. 立即废弃并轮换当前私钥。
  2. 私钥改为部署时注入，不再以文件形式进入仓库。
  3. `.gitignore` 增加密钥文件忽略规则。
  4. 检查 Git 历史、压缩包、发布包、聊天传输中是否已经扩散。

### F-02 `P0` 短信验证码开发绕过常开，并且接口直接返回验证码

- 风险等级：严重 / 最高优先级
- 位置：
  - `miniprogram/cloudfunctions/sendSmsCode/index.js:16`
  - `miniprogram/cloudfunctions/sendSmsCode/index.js:109-131`
  - `miniprogram/cloudfunctions/bindPhone/index.js:19`
  - `miniprogram/cloudfunctions/bindPhone/index.js:118-129`
- 现象：
  - `DEV_BYPASS.enabled` 写死为 `true`
  - 测试手机号命中后，`sendSmsCode` 直接返回 `devCode`
  - `bindPhone` 也允许固定验证码直通
- 风险说明：
  - 这相当于把“万能验证码后门”直接带进了正式代码。
  - 即使只对白名单手机号生效，也属于不可接受的生产风险。
  - 更严重的是，接口把 `devCode` 返回给前端，等于主动把验证码交给调用方。
- 建议：
  1. 默认改成关闭。
  2. 只允许在本地或明确的测试环境通过环境变量开启。
  3. 即使是测试环境，也不要把 `devCode` 回传给客户端。
  4. 绑定、换绑、后台登录三套验证码链路都要统一治理。

### F-03 `P1` 敏感信息被写入云函数日志

- 风险等级：高
- 位置：
  - `miniprogram/cloudfunctions/adminLogin/index.js:30`
  - `miniprogram/cloudfunctions/sendSmsCode/index.js:40`
  - `miniprogram/cloudfunctions/sendSmsCode/index.js:130`
  - `miniprogram/cloudfunctions/sendSmsCode/index.js:152`
  - `miniprogram/cloudfunctions/sendSmsCode/index.js:169`
  - `miniprogram/cloudfunctions/bindPhone/index.js:98`
  - `miniprogram/cloudfunctions/bindPhone/index.js:129`
  - `miniprogram/cloudfunctions/getMyStats/index.js:59`
  - `miniprogram/cloudfunctions/adminGetStats/index.js:63`
  - `miniprogram/cloudfunctions/adminGetStaffList/index.js:29`
- 现象：
  - `adminLogin` 直接记录完整入参，等价于把手机号和短信验证码写进日志。
  - `sendSmsCode` 记录手机号，开发绕过分支还记录了真实验证码。
  - `bindPhone` 记录手机号和 `openid`。
- 风险说明：
  - 日志系统常常会被运维、开发、排障脚本、导出包间接访问。
  - 一旦日志里出现验证码、手机号、`openid`，就会形成隐私与审计风险。
- 建议：
  1. 登录、验证码、绑定相关接口禁止记录完整入参。
  2. 统一做日志脱敏：手机号仅保留前三后四，验证码永不入日志。
  3. 只保留必要的排障字段，如接口名、成功失败、错误码、耗时。

### F-04 `P1` 富文本缺少服务端净化，管理端预览直接 `v-html`

- 风险等级：高
- 位置：
  - `admin/src/views/cases/edit.vue:151`
  - `admin/src/views/cases/list.vue:219`
  - `admin/src/components/TinyEditor.vue:212-213`
  - `miniprogram/cloudfunctions/adminCreateCase/index.js:61`
  - `miniprogram/cloudfunctions/adminUpdateCase/index.js`（接收并直接落库 `content`）
  - `miniprogram/cloudfunctions/adminUpdateCompany/index.js`（接收并直接落库 `intro` / `businessIntro`）
  - `miniprogram/cloudfunctions/getCardInfo/index.js:57-97,183-184`
  - `miniprogram/cloudfunctions/getCaseDetail/index.js:56-96,151`
- 现象：
  - 后端只做了图片地址和样式归一化，没有做真正的 HTML 安全净化。
  - 管理端案例预览直接把数据库里的 HTML 用 `v-html` 插入页面。
- 风险说明：
  - 现在的安全主要依赖 TinyMCE 编辑器白名单，但这只是前端限制，不是后端最终安全边界。
  - 一旦有人绕过编辑器，或者后续从数据库脚本/导入工具写入恶意 HTML，管理端预览就可能触发存储型 XSS。
- 建议：
  1. 在云函数落库前做服务端 HTML 白名单净化。
  2. 对 `href`、`src` 协议做限制，拒绝 `javascript:` 等危险协议。
  3. 管理端预览使用可信净化结果，不直接吃原始库内 HTML。

### F-05 `P1` 短信验证码校验没有绑定业务场景

- 风险等级：高
- 位置：
  - `miniprogram/cloudfunctions/bindPhone/index.js:48`
  - `miniprogram/cloudfunctions/updatePhone/index.js:26`
  - 对照：
    - `miniprogram/cloudfunctions/adminLogin/index.js:51`
    - `miniprogram/components/phone-verify/index.js:38`
    - `admin/src/components/PhoneVerify.vue:109`
- 现象：
  - `sendSmsCode` 发码时有 `scene`
  - 后台登录校验时也限定了 `scene: 'login'`
  - 但 `bindPhone` 和 `updatePhone` 校验时，只按 `phone + used:false` 查最新验证码，没有校验 `scene`
- 风险说明：
  - 这样会导致“同一个手机号，不同业务场景的验证码”边界变弱。
  - 例如某场景下发出的验证码，可能被另一场景复用。
- 建议：
  1. `bindPhone` 只认 `scene='bind'`
  2. `updatePhone` 只认 `scene='changePhone'`
  3. 所有验证码记录统一校验：手机号、场景、有效期、是否已用、锁定状态

### F-06 `P2` 依赖版本与运行来源不稳定

- 风险等级：中
- 位置：
  - `miniprogram/cloudfunctions/adminLogin/package.json:8`
  - `admin/src/components/TinyEditor.vue:171`
  - `admin/src/components/TinyEditor.vue:185`
- 现象：
  - `adminLogin` 云函数依赖 `@cloudbase/node-sdk: "latest"`
  - TinyMCE 与语言包在运行时从 `cdn.jsdelivr.net` 加载
- 风险说明：
  - `latest` 会让同一份源码在不同时刻装出不同依赖，构建不可复现。
  - 运行时 CDN 依赖会引入外网可用性问题，也增加供应链不确定性。
- 建议：
  1. 生产云函数全部改为固定版本。
  2. TinyMCE 静态资源优先本地化或明确锁定来源与回退方案。

### F-07 `P2` 统计与列表接口存在明显的全量扫描 / `N+1` 查询风险

- 风险等级：中
- 位置：
  - `miniprogram/cloudfunctions/adminGetStats/index.js:204-259`
  - `miniprogram/cloudfunctions/adminGetStats/index.js:110,146,177,234`
  - `miniprogram/cloudfunctions/getMyStats/index.js:112-140`
  - `miniprogram/cloudfunctions/adminGetStaffList/index.js:44-68`
- 现象：
  - `adminGetStats` 会先拉全量员工，再对每个员工重复查 7 天、15 天、30 天、总量、历史汇总。
  - `getMyStats` 把整个时间段日志和全部日汇总拉回内存再聚合。
  - `adminGetStaffList` 当前是全量取完再组装，不分页。
- 风险说明：
  - 现在数据量小还能跑，数据量一大就会变慢、超时、或者耗费更多云资源。
  - 这类问题在统计页最容易先爆出来。
- 建议：
  1. 建立按天/按员工的统计汇总表。
  2. 后台排行改聚合查询，不要按员工逐个 `count()`
  3. 员工列表改分页。
  4. 给统计查询补索引与 Explain 分析。
